# 需求处理流程规则

## 规则来源
用户明确指示：2026-02-26 07:34:49
"记住一个需求啊，就是记记起来，就是我提出一个需求，你要优先在你的技能库里面检索一下是否能完成的。然后如果没有的话，可以去那个 codeho codehop 上面去解，去网上去检索一下，有没有相关能使用的技能，然后再给我，你们再自己想。就是先先检索能不能有，然后再问我需不需要去网上找，然后再问我要不要去，要不要自己做，怎么样？"

## 核心规则
**3步处理流程**：
```
1. 技能库优先检索 → 2. 外部资源搜索 → 3. 自主开发决策
```

## 详细步骤

### 步骤1：技能库优先检索
#### 执行动作
1. **立即检查**: 收到需求后首先检查现有技能库
2. **范围**: 当前38个已安装技能
3. **方法**: 
   - 根据需求关键词匹配技能描述
   - 检查技能功能是否覆盖需求
   - 评估技能可用性和成熟度

#### 输出要求
- **能完成**: 列出相关技能和具体实施方案
- **部分完成**: 说明能完成的部分和缺失部分
- **不能完成**: 明确说明现有技能无法满足

#### 检查清单
- [ ] 是否已检查所有相关技能？
- [ ] 是否有技能能完全满足需求？
- [ ] 是否有技能能部分满足需求？
- [ ] 是否需要组合多个技能？

### 步骤2：外部资源搜索
#### 触发条件
- 步骤1检查结果为"不能完成"或"部分完成"
- 需要更多解决方案时

#### 搜索范围
1. **CodeHop/ClawHub**: 技能市场平台
2. **GitHub**: 开源项目和代码库
3. **技术社区**: 相关技术论坛和社区
4. **文档资源**: 官方文档和技术博客

#### 询问格式
"步骤1检查发现现有技能无法完全满足需求。是否需要去网上搜索相关解决方案？"

#### 用户选择
- **需要搜索**: 执行外部搜索并返回结果
- **不需要搜索**: 直接进入步骤3
- **其他指示**: 按用户具体指示执行

### 步骤3：自主开发决策
#### 触发条件
- 步骤1和步骤2都没有合适解决方案
- 用户明确需要定制开发

#### 评估内容
1. **技术可行性**: 需求是否技术上可实现
2. **开发复杂度**: 预计开发工作量和难度
3. **资源需求**: 需要的时间、技术、资源
4. **优先级评估**: 相对于其他需求的优先级

#### 询问格式
"经过前两步检查，没有找到现成解决方案。是否需要自主开发？如果需要，我将制定详细开发计划。"

#### 用户选择
- **需要开发**: 制定详细开发方案和计划
- **不需要开发**: 记录需求为点子，后续考虑
- **修改需求**: 根据可行性调整需求范围

## 规则示例

### 示例：抖音自动评论需求
#### 步骤1执行
1. **检查技能库**: 现有38个技能中搜索"抖音"、"评论"、"自动化"
2. **结果**: 没有直接相关的抖音评论技能
3. **输出**: "现有技能库中没有抖音自动评论相关技能"

#### 步骤2询问
"现有技能无法满足需求。是否需要去CodeHop等平台搜索抖音相关技能？"

#### 步骤3询问（如果需要）
"外部搜索也没有合适方案。是否需要自主开发抖音自动评论技能？"

## 规则好处

### 对用户的好处
1. **效率优先**: 优先使用现有资源，快速解决问题
2. **成本控制**: 避免不必要的重复开发
3. **透明可控**: 每个步骤都询问用户意愿
4. **知识积累**: 系统化积累解决方案

### 对系统的好处
1. **技能复用**: 最大化现有技能的价值
2. **资源优化**: 合理分配开发资源
3. **流程规范**: 标准化的需求处理流程
4. **持续改进**: 不断丰富技能库和解决方案

## 执行检查清单

### 每次需求处理前检查
- [ ] 是否已理解需求的完整含义？
- [ ] 是否已执行步骤1（技能库检索）？
- [ ] 检索结果是否清晰明确？
- [ ] 是否需要进入步骤2（外部搜索）？
- [ ] 用户是否已做出明确选择？
- [ ] 处理过程是否符合3步流程？

### 记录要求
1. **需求记录**: 每个需求都要记录处理过程
2. **决策记录**: 记录用户的每个选择决定
3. **结果记录**: 记录最终处理结果和方案
4. **学习记录**: 记录从处理过程中学到的经验

## 例外情况处理

### 紧急需求
- 如果需求紧急，可以简化流程
- 但仍需保持核心的3步思考逻辑
- 事后补充完整记录

### 简单需求
- 明显简单或重复的需求可以快速处理
- 但仍需检查技能库可用性
- 保持基本的流程记录

### 复杂需求
- 复杂需求需要更详细的每个步骤
- 可能需要多次往返确认
- 保持完整的沟通记录
