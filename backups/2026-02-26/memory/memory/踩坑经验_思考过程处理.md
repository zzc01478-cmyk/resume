# 踩坑经验：思考过程处理

## 问题描述
重复发送思考过程给用户，多次被提醒。

## 踩坑记录
- **2026-02-26 06:30:00**: 用户首次指示"思考过程不需要发给我"
- **2026-02-26 06:35:43**: 用户指出"你还在给我发思考过程"
- **2026-02-26 06:43:51**: 用户再次指出并要求总结踩坑经验

## 根本原因
1. 习惯性详细解释
2. 对"思考过程"定义不清晰
3. 检查清单执行不严格
4. 未完全适应新通信规则

## 提炼规范

### 规则名称
思考过程处理规范

### 触发条件
任何回复给用户前

### 执行动作
1. 检查回复是否包含思考过程
2. 如有思考过程，删除并写入配置文件
3. 只保留结果和必要信息发送给用户
4. 技术细节保存到 `config/thinking_log.md`

### 文件位置
- SOUL.md Communication Style 章节
- 本文件作为永久规范

## 执行检查清单
- [ ] 是否包含思考过程？ → ❌ 删除
- [ ] 是否只发送必要信息？ → ✅ 保留  
- [ ] 技术细节是否写入配置文件？ → ✅ 是
- [ ] 是否简洁直接？ → ✅ 是
- [ ] 是否符合用户明确要求？ → ✅ 是

## 思考过程定义
以下内容属于思考过程，不发送给用户：
- 解释性段落
- 分析推理过程
- 技术细节说明
- 调试信息
- 执行步骤详细描述

## 必要信息定义
以下内容可以发送给用户：
- 最终结果
- 任务状态
- 关键数据
- 行动确认
- 必要说明

## 配置文件结构
```
/root/.openclaw/workspace/config/
├── thinking_log.md          # 思考过程和技术细节
└── 其他技术配置文件
```

## 效果验证
- ✅ 用户只接收必要信息
- ✅ 回复更简洁高效
- ✅ 技术细节有完整记录
- ✅ 完全符合用户要求

## 学习点
1. 明确规则需要严格执行
2. 习惯改变需要时间适应
3. 以用户需求为最高优先级
4. 从错误中学习并改进

## 永久执行
本规范作为永久文件，避免重复踩坑。
